## 本輪修正

- LibreChat v0.8.5 實測仍需要 `RAG_API_URL`;原本未設定時上傳知識庫檔案會出現 `Error processing file`,LibreChat log 顯示 RAG API 不可達。
- 新增 `backend/accounting/routers/rag_adapter.py`,提供 LibreChat 相容 `/rag/health`、`/rag/text`、`/rag/embed`、`/rag/query`、`/rag/documents`,以 MongoDB 儲存抽取文字並用 deterministic keyword retrieval 做本地基準。
- `config-templates/docker-compose.yml` 將 LibreChat `RAG_API_URL` 指向 `http://accounting:8000/rag`,避免交付時多拉 upstream rag_api / pgvector registry image。
- `scripts/upload-knowledge-base.py` 改用 LibreChat v0.8 file_search multipart 欄位,並以精確名稱選 OpenAI 知識庫 Agent,避免模糊掛錯 Agent。
- nginx 已封鎖外部 `/api-accounting/rag/*`,RAG adapter 僅供容器內 LibreChat 使用。
- `/rag/embed`、`/rag/query`、`/rag/text`、`/rag/documents` 需 LibreChat short-lived JWT / internal token / dev legacy header,避免容器內未授權程序直接查詢或刪除 RAG 文件。

## 殘餘風險

- 目前 adapter 是交付安全的 keyword retrieval baseline,不是語意 embedding/pgvector；正式大量文件上線前仍建議以 3-5 份去識別化真實樣本重跑 Gate 2。
- 長 PDF/PPTX 仍有 `120000` 字元上限與頁碼 fidelity 限制;若公司導入大量標書,下一版要升級 chunk metadata / page extraction。
- Claude 知識庫 Agent 曾因 `thinking` + `temperature` 參數組合失敗,本次正式證據採 OpenAI 知識庫 Agent。
- 乾淨 Mac/VM 上仍需重跑本驗證並補截圖/錄影,本報告只證明本機 stack 的 LibreChat file_search 閉環已通。
