## 2. 可帶條件交付判定

### 2.1 可以交付的理由

- [ ] 系統方向清楚:承富 AI 是工作台,不是單純聊天機器人。
- [ ] 文件體系已形成:部署、訓練、維運、安全、Day 0、交付前 checklist 都已有基本骨架。
- [ ] 核心使用情境明確:投標、活動、設計、公關、營運後勤能對應承富日常工作。
- [ ] 已有驗收心智:採用率、first-win、Day 0 演練、乾淨環境驗收等都被納入交付思考。
- [ ] 已有風險意識:知道登入、真實資料、Champion、乾淨部署、備份還原是交付成敗關鍵。

### 2.2 不應無條件交付的理由

- [ ] 目前仍需要證明「從零環境」可成功安裝,不是只在開發機可跑。
- [ ] UI/UX 對 AI 小白仍可能太複雜,需要用真實使用者 first-win 驗證。
- [ ] 真實承富資料灌入與檢索品質尚需現場確認。
- [ ] Day 0 登入流程若失敗,會直接破壞信任。
- [ ] 採用成效尚未經過分階段 pilot 量測。

### 2.3 條件式交付門檻

達到以下條件才可進入正式 Day 0:

- [ ] 6 個必修項全部完成。
- [ ] 乾淨 Mac/VM 驗收通過。
- [ ] Champion 能獨立完成至少 3 個端到端案例。
- [ ] 至少 5 份代表性知識庫文件已上傳並可被引用。
- [ ] 10 人帳號流程已 dry-run,且不包含任何明文永久密碼交付。
- [ ] 老闆 / Champion 知道這是條件式上線,並接受第 1 週仍會調整。

## 4. 5 個高價值 Findings

### Finding 1:產品價值已接近可交付,但信任建立比功能數量更重要

系統已經不只是功能堆疊,但承富同仁是否願意使用,取決於前 5 分鐘能否成功登入、看懂入口、完成一個真實任務。Day 0 應優先追求 first-win,不要先介紹完整功能地圖。

建議行動:

- [ ] Day 0 先做角色任務,後講 UI。
- [ ] 老闆看到的是成果數字,不是功能清單。
- [ ] Champion 先拿出 3 個已成功案例,降低全員不確定感。

### Finding 2:UI/UX 最大風險不是美觀,而是心智負荷

目前系統功能多,但 AI 小白不應被迫理解 Agent、Workspace、RAG、MCP、model 等概念。首頁應該更像「今天要完成什麼工作」,而不是「有哪些工具可以選」。

建議行動:

- [ ] 用「任務 / 工作包 / 下一步 / 交付品」語言包裝。
- [ ] 弱化技術詞與助手選擇。
- [ ] 保留 Workspace,但避免變成大型分類選單。

### Finding 3:乾淨環境驗收是交付可信度核心

如果只在開發機可跑,就不能視為已完成交付。乾淨 Mac/VM 驗收能揭露缺少的 env、腳本依賴、Docker 設定、文件漏步與路徑假設。

建議行動:

- [ ] 交付前一定要做乾淨 Mac/VM 驗收。
- [ ] 驗收過程只照文件操作。
- [ ] 所有「需要口頭補充」的步驟都要回寫文件。

### Finding 4:真實資料與範例案例決定老闆是否覺得值得

沒有承富真實資料,系統很容易看起來像一個漂亮但空泛的 AI 入口。最有說服力的是讓它處理承富過往標案、結案、新聞稿、設計 brief、廠商信件。

建議行動:

- [ ] 交付前準備 5-10 份代表性文件。
- [ ] 每個展示案例都要能產出可保存交付品。
- [ ] 回答必須能指向資料來源或清楚說明資料不足。

### Finding 5:應採分階段試用,不要把 Day 0 當最終驗收

10 人一次全量上線風險高。比較穩妥的方式是先讓 Champion 與 2-3 位高頻使用者跑出成果,再擴到全員。Day 0 是啟動儀式與 first-win 驗收,第 1 週才是真正採用觀察。

建議行動:

- [ ] 先 Champion pilot。
- [ ] 再 3-5 人小組試用。
- [ ] 最後全員 Day 0。
- [ ] Day +1 到 Day +7 每天看使用數、卡點、first-win、滿意度。

## 5. 分階段試用建議

### Stage 0:乾淨環境與交付文件驗收

建議時間:0.5-1 天。

目標:

- [ ] 確認系統能在乾淨 Mac/VM 從零啟動。
- [ ] 確認文件足以讓非原開發者照做。
- [ ] 確認不需要任何未記錄的手動秘技。

通過條件:

- [ ] 乾淨 Mac/VM checklist 全部通過。
- [ ] 缺漏文件已補齊。
- [ ] 未發現阻斷性部署問題。

### Stage 1:Champion Pilot

建議時間:1-2 天。

目標:

- [ ] Champion 熟悉系統。
- [ ] Champion 跑完 3 個承富端到端案例。
- [ ] Champion 可協助 Day 0 登入與基礎操作。

通過條件:

- [ ] Champion 可獨立示範投標、設計或公關任一主流程。
- [ ] Champion 能說出 3 個常見卡點與解法。
- [ ] Champion 確認 Day 0 教學素材可用。

### Stage 2:小組 Pilot

建議時間:3-5 個工作日。

目標:

- [ ] 選 3-5 位高頻使用者試用。
- [ ] 每人完成至少 1 個真實 first-win。
- [ ] 收集 UI/UX、AI 回答、資料來源、流程中斷點。

通過條件:

- [ ] 至少 3 人完成真實工作成果。
- [ ] 至少 2 個成果可給老闆展示。
- [ ] 沒有登入、資料灌入或核心流程阻斷問題。

### Stage 3:全員 Day 0

建議時間:1 天。

目標:

- [ ] 10 人完成登入。
- [ ] 至少 7 人完成 first-win。
- [ ] 角色覆蓋 PM、設計、業務 / 公關。

通過條件:

- [ ] 登入成功率 >= 9/10。
- [ ] first-win 完成率 >= 7/10。
- [ ] 老闆與 Champion 知道第 1 週仍是觀察與微調期。

### Stage 4:Day +1 到 Day +7 採用觀察

建議時間:交付後第一週。

目標:

- [ ] 每天觀察用量、滿意度、卡點。
- [ ] Champion 每天回報前 3 個問題。
- [ ] Sterio 修正阻斷性文件或流程問題。

通過條件:

- [ ] 至少 5/10 同仁在第一週內完成第二次使用。
- [ ] 至少 3 個真實成果被保存或交付。
- [ ] 沒有需要 rollback 的阻斷問題。

## 6. 乾淨 Mac/VM 驗收 Checklist

> 原則:驗收者只能使用 repo 內文件與公開工具安裝流程,不得依賴開發機既有資料、shell history、私人暫存檔或未記錄的本機設定。

### 6.1 驗收環境

- [ ] 使用乾淨 Mac 或乾淨 VM。
- [ ] OS 版本記錄完成。
- [ ] Docker Desktop 或等效 Docker runtime 已安裝。
- [ ] Git 已安裝。
- [ ] Python / Node / shell 工具版本符合文件需求。
- [ ] 本機沒有預先存在的 ChengFu 容器、volume、database 或 repo checkout。
- [ ] 驗收記錄不包含任何 secret 值。

### 6.2 原始碼與文件

- [ ] 從 repository 重新 clone。
- [ ] 先讀 `AGENTS.md`、`docs/DECISIONS.md`、`DEPLOY.md`、`ARCHITECTURE.md`。
- [ ] 確認 `config-templates/.env.example` 可作為填寫依據。
- [ ] 確認文件明確說明哪些值必須由 Keychain / 環境變數提供。

### 6.3 Secret 與設定處理

- [ ] 不把任何真實 secret 寫進 repo。
- [ ] 不把任何真實 secret 寫進本文件。
- [ ] 使用 Keychain 或本機安全方式注入必要 secret。
- [ ] `.env` 僅包含非機密或 placeholder。

### 6.4 啟動驗收

- [ ] 依文件執行啟動腳本。
- [ ] nginx container 啟動。

- [ ] MongoDB container 啟動。

### 6.5 核心功能驗收

- [ ] Launcher 首頁可載入。

- [ ] 深 / 淺色模式不破版。
- [ ] LibreChat 入口可開啟。
- [ ] 建立或確認核心 Agent / 助手流程可執行。
- [ ] 知識庫上傳流程可執行於測試文件。
- [ ] 測試查詢能回覆並指向測試資料來源。

### 6.6 測試與 Smoke

- [ ] 執行 smoke test。
- [ ] 執行後端測試或文件指定的最小測試集。
- [ ] 若有 Playwright / E2E 測試,執行交付前指定 subset。

### 6.7 備份與還原

## 7. 交付決策建議

建議目前決策為:

> **可帶條件交付。**  
> 在 6 個必修項與乾淨 Mac/VM 驗收完成前,不建議對承富宣稱正式 production-ready。完成後可進入 Champion Pilot,再擴大到小組 Pilot 與 Day 0 全員 first-win。

不建議的做法:

- [ ] 不要跳過乾淨 Mac/VM 驗收。
- [ ] 不要只展示 demo UI 就簽收。
- [ ] 不要 Day 0 一開始講完整功能。
- [ ] 不要在沒有真實資料的情況下要求老闆判斷 ROI。
- [ ] 不要把第 1 週採用問題視為單純教育訓練問題;它也可能是產品流程問題。
